# Bug Report — Brand Module

## BUG-BR-001: Route `brands/reorder` Conflict With `apiResource`

**Severity:** Low

**Component:** `packages/marvel/src/Rest/Routes.php` (lines 680-681)

**Description:** The `PUT brands/reorder` route is defined before `apiResource('brands', ...)`. This is currently correct — the literal route matches before `{brand}` parameter binding. However, if the route order were ever changed (e.g., alphabetically sorted or reorganized), `PUT brands/reorder` would be caught by `PUT brands/{brand}` → `update()` and would attempt to find a brand with ID "reorder", resulting in a 404 instead of the reorder functionality.

**Code Location:** `packages/marvel/src/Rest/Routes.php` — lines 680-681

**Current Behavior:**
```php
Route::put('brands/reorder', [BrandController::class, 'reorder']);  // line 680
Route::apiResource('brands', BrandController::class);                // line 681
```

**Risk:** Currently low — no bug exists. But this is a fragile ordering dependency that could break during refactoring.

**Recommendation:** Add a defensive comment explaining the ordering requirement, or use a different URL pattern (e.g., `PUT brands/reorder/{ids}` or `POST brands/reorder`).

## BUG-BR-002: No Guard Against Reordering Non-Existent Brands

**Severity:** Low

**Component:** `Marvel\Http\Requests\BrandsReorderRequest`

**Description:** The validation rule `brands.*` => `exists:brands,id` only checks that each brand ID exists at the time of validation. There is a race condition: if a brand is deleted between validation and the `setNewOrder()` call, the order will silently skip the missing brand.

**Code Location:** `BrandsReorderRequest::rules()`, `BrandRepository::reorder()`

**Impact:** Low — the window is very small. The `setNewOrder()` method from Spatie SortableTrait ignores non-existent IDs.

## BUG-BR-003: Slug Not Generated When Name Is Not Dirty

**Severity:** Low

**Component:** `Brand` model `saving` event

**Description:** The `saving` event only generates a slug when `name` is dirty and `slug` is not dirty. If a brand is created without a name but with a slug (e.g., via import), the slug is preserved. This is the intended behavior, but if the name is later updated without changing the slug, the slug regenerates from the new name.

**Code Location:** `packages/marvel/src/Database/Models/Brand.php` — `booted()` method

**Current Behavior:**
```php
static::saving(function ($brand) {
    if ($brand->isDirty('name') && !$brand->isDirty('slug')) {
        $enName = $brand->getTranslation('name', 'en', false);
        $brand->slug = $enName ? Str::slug($enName) : null;
    }
});
```

**Impact:** Low — this is the expected behavior for auto-slug generation.

## BUG-BR-006: `BrandUpdateRequest` Ignores Own Name via Route Parameter

**Severity:** Low

**Component:** `Marvel\Http\Requests\BrandUpdateRequest`

**Description:** The uniqueness check uses `$this->route('brand')` to get the current brand ID for the ignore rule. If the route parameter name changes (e.g., from `brand` to something else), the ignore logic would silently fail, and the user would get a false "name already exists" error when updating their own brand.

**Code Location:** `packages/marvel/src/Http/Requests/BrandUpdateRequest.php`

**Current Behavior:**
```php
$id = $this->route('brand');
```

**Impact:** Low — the route parameter name is stable as it's generated by `apiResource('brands', ...)` which consistently uses singular snake_case (`brand`).

## BUG-BR-007: No Transaction on `reorder()` Operation

**Severity:** Low

**Component:** `Marvel\Database\Repositories\BrandRepository::reorder()`

**Description:** The `reorder()` method calls `$this->setNewOrder($brands)` without wrapping in a database transaction. If the operation fails mid-way, some brands may have updated order values while others remain at their previous positions.

**Code Location:** `packages/marvel/src/Database/Repositories/BrandRepository.php`

**Current Behavior:**
```php
public function reorder(array $brands)
{
    try {
        $this->setNewOrder($brands);
    } catch (\Exception $e) {
        throw new HttpException(500, $e->getMessage());
    }
}
```

**Recommended:**
```php
public function reorder(array $brands)
{
    try {
        DB::transaction(function () use ($brands) {
            $this->setNewOrder($brands);
        });
    } catch (\Exception $e) {
        throw new HttpException(500, $e->getMessage());
    }
}
```

**Impact:** Low — `setNewOrder()` is a simple column update that rarely fails.
